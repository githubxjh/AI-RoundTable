# CLAUDE.md

这是给 Claude 和其他 coding agent 的行为约束。和 `AGENTS.md` 一起看；如果有冲突，以更具体、更新的用户指令为准。

## 1. 先想清楚再写

不要默认，不要装作已经明白。

动手前先做这几件事：

- 说清楚当前目标和成功标准。
- 把关键假设写出来。
- 如果有多个理解方式，直接摆出来，不要暗自挑一个。
- 如果有更简单的方案，要说明。
- 如果关键要求不清楚，就先停下来问。

## 2. 先简单

只写刚好够用的东西，不做多余设计。

- 不做需求之外的功能。
- 不为单次使用去造抽象。
- 不加用户没要求的灵活配置。
- 尽量沿用现有风格，而不是按“我更喜欢的风格”重排。
- 如果改动变大了，先想有没有更小的行为等价路径。

## 3. 外科式改动

只改必须改的地方，自己产生的脏东西自己清。

- 不要顺手优化旁边的代码、注释或格式。
- 不要重构无关模块。
- 对于自己改动后变成未使用的 import/变量/函数，要顺手删掉。
- 发现无关死代码可以说出来，但不要擅自删。

检验标准：每一行改动都能直接对应到用户这次请求。

## 4. 以目标驱动执行

把任务转成可验证目标：

```text
加校验 -> 写/跑无效输入检查，然后让它通过。
修 bug -> 先复现或隔离，再验证修复。
重构 -> 先有行为覆盖，再改结构。
```

多步骤任务先用一个短计划：

```text
1. 步骤 -> 验证: 检查点
2. 步骤 -> 验证: 检查点
3. 步骤 -> 验证: 检查点
```

## 5. AI-RoundTable 约定

- 做大一点的工作前，先读 `AGENTS.md`、`.claude/STATUS.md`、`.claude/HANDOFF.md`、`TESTING.md` 和 `package.json`。
- Windows 上统一用 `cmd /c npm.cmd ...`。
- ChatGPT、Claude、Grok、Gemini、Doubao、DeepSeek 这些模型名保持英文。
- Lite/public 和 Advanced/local 的附件路径要分开看。
- 真正的浏览器检查优先走 `TESTING.md` 和 `docs/self-iteration.md` 里写的 attach-mode 流程。
- 遇到登录、验证码、人工验证、2FA 或账号阻断时，要停下来并明确说明卡在哪里。
