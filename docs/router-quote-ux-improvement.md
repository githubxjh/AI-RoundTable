# 路由引用机制的 UX 改进方案

## 本轮执行锁定

> 文档状态：本轮只优化方案文档，不改代码。下面提到的代码、测试和文件路径只作为接手 AI 的实现边界。

### 推荐方案

固定采用 **方案二：Toast + 行内禁用原因说明**：

- `导入评审` 成功后用 Toast 告知“已导入哪个模型的答案 + 几条外部评审意见”。
- 在路由目标区上方常驻显示行内提示，明确“已引用哪些模型、为什么这些模型不能作为路由目标、如何恢复”。
- 对因引用而禁用的目标加可见样式和 `title`，与“模型本身不可用”的禁用状态区分开。

### 不要做什么

- 不要修改“引用模型不能路由给自己”的业务规则。
- 不要重构整个路由目标 UI 分组；本轮不做方案三。
- 不要改 `router_review_import.mjs` 的引用数据构建逻辑。
- 不要只依赖 hover tooltip；原因必须在面板里持久可见。

### 实现注意事项

- 行内提示至少包含三类信息：已引用模型列表、禁用原因、恢复方法。
- 文案示例：`已引用 ChatGPT、Grok 的回答，这些模型不能作为路由目标；如需发送给它们，请先移除对应引用。`
- 引用数量徽章和导入后的短暂高亮属于推荐增强项，但不能替代行内原因说明。
- 所有新增文案同步走 `_locales/zh_CN/messages.json` 和 `t()`。
- 验收由代码执行方完成；本文档只记录测试要求，不在本轮执行测试。

## 问题诊断

### 用户反馈

> "有些用户不知道默认引用了，不能再发送，他会以为是 bug"

### 根因分析

问题出在「导入评审 → 自动填充引用列表 → 路由目标被禁用」这条链路，每一步对用户都是黑盒。

#### 触发链路

```
用户点击"导入评审"
  → onImportCandidateReview() [panel.js:1158]
    → buildReviewImportBundle() [router_review_import.mjs:3]
      → 自动构建 answerQuote（候选答案原文）
      → 自动收集 feedbackQuotes（各评委的评审意见）
      → 返回 quoteList，通常包含 2-4 条引用
    → state.quoteList = bundle.quoteList [panel.js:1187]  ← 静默覆盖
    → refreshRouterComposer()
      → updateRouteExclusions() [panel.js:965]
        → quotedSources = new Set(state.quoteList.map(item => item.source))
        → 遍历路由目标 checkbox，命中 quotedSources 的 → disabled + unchecked
```

#### 三个具体问题

| # | 问题 | 用户感知 |
|---|------|---------|
| 1 | **导入评审静默填充引用列表** | 用户不知道引用列表变了，更不知道为什么变了 |
| 2 | **被引用模型的路由目标被禁用，无解释** | checkbox 灰掉、勾选消失，用户以为是功能坏了 |
| 3 | **"引用 = 不能路由给自己"的规则从未被说明** | 即使用户注意到禁用了，也不理解背后的逻辑 |

#### 代码位置

- 引用列表渲染：[panel.js:866-892](src/sidepanel/panel.js#L866)
- 路由目标禁用逻辑：[panel.js:992-1007](src/sidepanel/panel.js#L992)
- 导入评审入口：[panel.js:1158-1192](src/sidepanel/panel.js#L1158)
- 引用数据结构构建：[router_review_import.mjs](src/sidepanel/router_review_import.mjs)

## 方案

### 方案一：最低成本——加 Toast + title 属性

**改动：** 3 处，约 15 行代码

1. `onImportCandidateReview` 成功后，弹一条 Toast：*"已导入 X 的答案及 N 条评审意见到引用列表"*
2. `updateRouteExclusions` 中，给因引用而禁用的 checkbox 的父 label 加 `title`：*"该模型已在引用列表中，无法路由给自己"*
3. 引用列表为空时，已有的提示文案保留。

**优势：**
- 改动极小，几乎零风险
- 用户至少能感知到"导入 → 引用列表变了"
- 鼠标悬停禁用目标可看到原因

**劣势：**
- 用户如果不 hover 禁用的 checkbox，仍然看不到原因
- Toast 自动消失后，后来才看路由面板的用户没有持久提示
- 没有从根本上让规则变清晰，只是加了事后解释

**适用判断：** 快速止血。如果只是想消除"以为是 bug"的困惑，这个方案能解决 70%。

---

### 方案二：中等投入——Toast + 禁用原因行内展示

**改动：** 5-6 处，约 40 行代码

在方案一基础上增加：

1. **引用列表区域加数量徽章**：在路由面板的"引用列表"标题旁显示引用数量，导入后有短暂高亮动画，让用户注意到变化。

2. **路由目标区域加禁用原因说明**：在 `.router-targets` 上方，当存在因引用被禁用的模型时，显示一行灰色小字：*"已引用 ChatGPT、Grok 的回答，这些模型无法路由给自己——如需路由到它们，请先在引用列表中移除对应条目。"*

3. **被禁用的 checkbox label 加删除线样式**：视觉上区分"模型本身不可用"和"因引用而暂时不可用"。

**优势：**
- 原因持久可见，用户不需要 hover
- 给出了解决方法（去引用列表移除），形成闭环
- 视觉上区分了两种禁用（模型不可用 vs 被引用占用），减少困惑

**劣势：**
- 需要监听引用列表变化来更新禁用说明
- 如果引用列表很长，提示文字会变长

**适用判断：** 投入产出比最好的方案，推荐。

---

### 方案三：深度优化——重构路由目标 UI

**改动：** 涉及 HTML + CSS + JS，约 100+ 行代码

将路由目标的 checkbox 列表改为分组展示：

```
可选目标
☑ ChatGPT   ☐ Grok   ☐ Doubao

已引用（不可路由给自己）
⊘ Gemini（你的引用列表中包含 Gemini 的回答）
⊘ DeepSeek（你的引用列表中包含 DeepSeek 的回答）
```

同时：
- 导入评审前弹出简要确认："将把 X 的回答及 N 条评审意见加入引用列表，这些模型将暂时无法作为路由目标。"
- 引用列表项旁加"移除并恢复路由目标"快捷按钮
- 支持一键清空引用列表并重置路由目标勾选

**优势：**
- 信息架构清晰，一眼看懂哪些可选、哪些不行、为什么
- 操作路径短（移除引用 → 自动恢复目标）
- 导入前的确认消除意外感

**劣势：**
- 改动量大，HTML 结构需要调整
- 需要新增交互逻辑，回归测试范围大
- 对于使用频率不高的功能，性价比存疑

**适用判断：** 如果路由是这个产品的核心高频功能，值得投入。如果只是辅助功能，方案二足够。

## 方案对比

| 维度 | 方案一 Toast+title | 方案二 Toast+行内说明 | 方案三 重构 UI |
|------|:---:|:---:|:---:|
| 实现工时 | 0.5h | 1-2h | 4-6h |
| 改动文件 | 1 (panel.js) | 2 (panel.js + panel.html) | 3+ (js+html+css) |
| 用户能发现原因 | 需 hover | 直接可见 | 一目了然 |
| 给出解决方法 | ❌ | ✅ | ✅ |
| 区分两种禁用 | ❌ | ✅ 样式区分 | ✅ 分组展示 |
| 导入前告知 | ❌ | ❌ | ✅ |
| 回归风险 | 极低 | 低 | 中 |

## 建议

推荐**方案二**。理由：

1. 1-2 小时工时，改动集中在两个文件
2. 解决了核心矛盾：用户能看到"为什么被禁用"+"怎么解决"
3. 行内说明持久存在，不像 Toast 会消失，也不像 tooltip 需要主动发现
4. 如果后续觉得不够，可以在方案二基础上平滑升级到方案三（分组展示）

## 实现要点（给接手 AI 的 brief）

### 必须改的

1. **`onImportCandidateReview` 成功路径**（[panel.js:1187](src/sidepanel/panel.js#L1187) 之后）：Toast 提示"已导入 X 的答案及 N 条评审意见"。

2. **`updateRouteExclusions`**（[panel.js:1000-1007](src/sidepanel/panel.js#L1000)）：给被引用禁用的 checkbox 父 label 添加 `data-quoted="true"` 属性和 `title` 说明。

3. **`.router-targets` 上方**（[panel.html:866](src/sidepanel/panel.html#L866) 之前）：插入一个 `<div id="route-targets-hint">`，动态显示哪些模型因引用被禁用。

### 建议改的

4. **CSS**：`label[data-quoted="true"]` 加删除线或半透明样式，与 `label.disabled`（模型本身不可用）区分开。

5. **`renderQuoteList`**（[panel.js:866](src/sidepanel/panel.js#L866)）：引用列表标题旁显示数量徽章，导入后有短暂过渡动画（CSS transition）。

### 不要改的

- 不要改"引用 = 不能路由给自己"这条业务规则本身——它在逻辑上是合理的（防止循环路由），问题只在 UI 层没有传达。
- 不要改 `router_review_import.mjs` 的数据构建逻辑。
