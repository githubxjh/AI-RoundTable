# 浏览器原生弹窗替换方案

## 本轮执行锁定

> 文档状态：本轮只优化方案文档，不改代码。下面提到的代码、测试和文件路径只作为接手 AI 的实现边界。

### 推荐方案

固定采用 **方案二：Toast + Modal**：

- 成功反馈和校验拦截使用自定义 Toast。
- 二次确认使用自定义 Modal，保留明确的确认/取消动作。
- Toast 和 Modal 都用原生 HTML/CSS/JavaScript 实现，符合项目 Vanilla 架构。

### 不要做什么

- 不要把 `confirm` 场景改成“Toast + 撤销”，确认感太弱。
- 不要引入第三方 Toast、Modal、Headless UI 或 React/Vue 组件库。
- 不要照搬 `reference/chathub` 的 UI 组件代码；最多参考交互语义。
- 不要在本轮推进 Inline 表单错误体系，避免把弹窗替换扩散成大范围表单重构。

### 实现注意事项

- 替换 `window.alert` / `window.confirm` 时，必须同步更新 smoke 测试里等待浏览器 `dialog` 的断言；否则弹窗被正确替换后，测试会因为仍在等待原生 dialog 而假失败。
- 推荐先落地 Toast，再落地 Modal；每一步替换后都清点剩余 `window.alert` / `window.confirm`。
- 所有新提示文案走 `t()` 和 `_locales/zh_CN/messages.json`，不要只写死在 `panel.js` 或 `panel.html`。
- 验收由代码执行方完成；本文档只记录测试要求，不在本轮执行测试。

## 现状

`src/sidepanel/panel.js` 中共有 25 处原生浏览器弹窗调用：

- **window.alert** × 20 — 阻塞式提示
- **window.confirm** × 5 — 阻塞式确认

问题：原生弹窗样式不可控、阻塞 JS 主线程、浏览器会显示"阻止此页创建其他对话框"的复选框、用户体验差。

## 弹窗场景分类

按语义分为三类：

| 类别 | 数量 | 典型场景 |
|------|------|---------|
| 成功反馈 | 6 | 路由已发送、轮次已删除、评审已启动 |
| 校验拦截 | 14 | 输入为空、模型禁用、候选不足 |
| 二次确认 | 5 | 删除轮次、覆盖评审结果 |

---

## 方案一：纯 Toast（最简）

**做法：** 所有 alert 和 confirm 全部改用非阻塞 Toast 通知，confirm 场景额外加一个"撤销"按钮。

**改动量：** 小。引入一个 Toast 库（或手写 50 行），全局替换 `window.alert` / `window.confirm` 为 `toast.show()` / `toast.confirm()`。

**优势：**
- 实现最简单，改动最集中
- 不阻塞 UI，用户体验改善明显
- 浏览器不会再弹出"阻止对话框"提示

**劣势：**
- confirm 改为 Toast 后，确认感弱——用户可能在没注意到 Toast 的情况下误操作
- 校验拦截用 Toast 提示，错误信息与触发位置分离，关联性弱
- 不解决"输入前就应该知道规则"的问题（如评分至少需要 2 个候选）

**适用判断：** 如果你想快速消除浏览器弹窗且能接受 confirm 变弱，这是最快路径。

---

## 方案二：Toast + Modal（分治）

**做法：** 按场景分类处理——

- 成功反馈 → Toast（2 秒自动消失，绿色图标）
- 校验拦截 → Toast（红色/黄色，可带操作按钮）
- 二次确认 → 自定义 Modal（居中弹层，有"确认/取消"按钮，样式统一）

自带 Dialog 组件在 `reference/chathub/src/app/components/Dialog.tsx`（基于 Headless UI），可参考或复用。

**改动量：** 中。需引入 Toast 组件（~100 行）、封装一个 confirm Modal（~80 行），然后逐处替换。

**优势：**
- 三种场景各有合适的交互模式，体验最佳
- Modal 保留确认感，防止误操作
- Toast 不打断流程，信息反馈自然

**劣势：**
- 需要维护两个新 UI 组件
- 替换时需要逐一判断每条调用属于哪类，人工决策成本略高
- Modal 需要管理 open/close 状态，改造成本比纯 Toast 高

**适用判断：** 如果你想做到位且不在乎多写一点 UI 代码，这是推荐方案。

---

## 方案三：Toast + Inline（激进优化）

**做法：** 在方案二基础上，进一步把校验拦截从 Toast 改为**内联提示**——

- 输入为空 → 输入框下方红色文字 + 边框变红
- 未选模型 → 选择区域高亮提示
- 候选不足 → 按钮旁显示剩余所需数量

成功反馈仍然用 Toast，二次确认仍然用 Modal。

**改动量：** 大。需要在多个 DOM 位置插入错误提示元素，为每个校验场景写对应的 UI 逻辑。

**优势：**
- 错误信息就地呈现，关联性最强，用户不用找"哪里错了"
- 可以在用户操作前就展示约束（如"至少需要 2 个候选"常驻显示）

**劣势：**
- 改动分散在面板各处，测试范围大
- 部分校验在 JS 逻辑深处触发（如路由响应返回错误），并非前端表单校验，内联改造成本高
- 过度设计风险——当前项目不是表单密集型应用

**适用判断：** 如果你愿意投入较多时间打磨细节，且认为当前提示方式严重影响使用效率。大多数场景下方案二已足够。

## 方案对比

| 维度 | 方案一 Toast | 方案二 Toast+Modal | 方案三 Toast+Inline+Modal |
|------|:-----------:|:-----------------:|:------------------------:|
| 实现工时 | 0.5 天 | 1 天 | 2-3 天 |
| 改动文件数 | 1-2 | 2-3 | 5+ |
| 阻断弹窗消失 | ✅ | ✅ | ✅ |
| 确认感保留 | ❌ 弱 | ✅ | ✅ |
| 错误关联性 | 一般 | 一般 | 强 |
| 回归风险 | 低 | 低 | 中 |
| 长期可维护性 | 好 | 好 | 一般（分散） |

---

## 建议

推荐**方案二（Toast + Modal）**。理由：

1. 1 天工时，投入产出比最高
2. 分类清晰：成功=Toast、确认=Modal、校验=Toast（红色），不需要像方案三那样侵入各处 DOM
3. 消除所有 `window.alert` / `window.confirm`，浏览器不再弹出"阻止对话框"提示
4. 后续如果你想进一步做内联提示，可以在方案二基础上渐进增强

---

## 实现要点（给接手 AI 的 brief）

1. **Toast 组件**：固定在右上角，支持 `success` / `error` / `warning` 三种类型，自动消失（success 2s，error/warning 4s），支持手动关闭，支持堆叠显示。

2. **Confirm Modal**：封装一个函数 `showConfirm({ title, message, onConfirm, onCancel })`，返回 Promise，调用方式接近 `window.confirm` 的同步风格，降低替换成本。

3. **替换顺序**：建议先替换所有 `window.alert` → Toast（最快见效），再逐个替换 `window.confirm` → Modal。
