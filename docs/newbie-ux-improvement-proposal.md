第一轮只执行 **方案 A + 方案 D 的轻量版本**：

- 如果做可关闭的新手提示，状态存储必须沿用现有 `RT_KEYS` / `rt_settings` 风格，不新增随意命名的 storage key。

## 当前问题诊断

经过对插件 UI 和信息架构的审查，梳理出以下具体问题：

### 问题 1：术语"路由器"（Router）晦涩难懂

`路由器` 这个名称来自原 PRD 中的技术概念 `Router Dock`（路由区），但对普通用户来说，"路由"在网络语境中指的是 WiFi 路由器，放到 AI 对话工具里会让人困惑——"这是干嘛的？跟网络有什么关系？"

**实际功能**：将某个 AI 的回答"引用"过来，加上你的新指令（如"帮我找找这段分析的漏洞"），再发送给另一个 AI 进行分析或辩论。本质是**"跨模型转发 + 追加指令"**，类似于把 A 的观点拿给 B 审阅。

### 问题 2："主任务"和"修饰器"的 Chip 标签缺少解释

路由器内部有分成两行的 Chip 按钮：
- **主任务**：找漏洞 / 核事实 / 看取舍 / 做决策 / 合并稿 / 回应评审
- **修饰器**：补盲区 / 提问题 / 要落地

问题在于：
- 用户不知道这些 Chip 是什么——它们是"一键生成 AI 提示词"的快捷方式
- 用户不知道该先选主任务再选修饰器的交互顺序
- "修饰器"这个名称本身是编程术语，非技术用户看不懂
- 每个 Chip 点击后到底会生成什么指令，用户只能瞎猜

### 问题 3："评审"区的"模式"和"标签"选项含义不明

评审区域有两个下拉框：
- **模式**：评分评审 / 讨论评审
- **标签**：匿名（A/B/C/D）/ 真实模型名

小白的困惑：
- "评分评审"和"讨论评审"到底有什么区别？什么场景该用哪个？
- "标签"是干什么的？为什么要匿名？
- 评委模型已经勾选了，为什么还要选模式和标签？
- 整个评审流程是什么：点了"开始评审"之后会发生什么？

### 问题 4：整体信息架构缺乏引导

- 界面从上到下是：全局提问 → 模型卡片 → 轮次信息 → 路由器 → 评审区，但没有任何一个区域解释**"这块是干什么的"**
- 新人打开插件后看到的是一堆按钮、复选框、下拉框，没有操作指引
- "轮次"的概念（一轮 = 一次广播 + 收集所有候选答案）完全不解释
- "加入候选"按钮的含义（"把这句回答纳入当前轮次，作为后续评审的素材"）也没有说明

## 方案对比

### 方案 A：轻量文案优化（推荐优先执行）

**核心思路**：不改 UI 结构，只改命名和增加辅助文字。

**具体改动**：

| 项目 | 当前文案 | 改为 |
|------|---------|------|
| 区域标题 | 路由器 | **交叉辩论** 或 **观点互评** |
| 主任务标签 | 主任务 | **我想让另一个 AI...** |
| 修饰器标签 | 修饰器 | **再额外要求...** |
| 主任务 Chip | 找漏洞 / 核事实 / ... | 保持，但 hover 时 tooltip 显示完整指令预览 |
| 评审-模式 | 模式 | **评审方式** |
| 评审-标签 | 标签 | **答案署名** |
| 按钮 加入候选 | 加入候选 | **纳入本轮** |
| 按钮 群发 | 群发 | **同时提问** |

同时新增：
- 每个区域标题旁加一个 `?` 图标，hover 弹 tooltip 解释该区域功能（纯 CSS title 属性实现，零依赖）
- 空状态提示文案优化（如"点击模型卡片的"引用"按钮，把回答加入此处，就可以让其他 AI 来审视它了"）
- 评审区新增一行说明文字："让多个 AI 相互打分/讨论，帮你选出最佳答案"

**优点**：
- 改动量极小，不涉及 JS 逻辑变更，基本只改 HTML 和 i18n 文案
- 风险极低，不会引入 bug
- 可以快速上线验证效果

**缺点**：
- 对完全不看文字的用户无效
- 没有从根本上改变用户的心智模型
- 空间有限的 tooltip 无法承载复杂解释

### 方案 B：分步引导 + 折叠面板（中等改动）

**核心思路**：将界面按使用流程分阶段呈现，新用户看到的不是完整面板，而是按步骤解锁。

**具体改动**：

1. **首次使用弹窗**：插件首次安装/更新后，弹出一个简洁的引导卡片（3 步）：
   - 第 1 步：在顶部输入问题，勾选 AI 模型，点击"同时提问"
   - 第 2 步：看到回答后，点击"引用"把你觉得好的/有疑问的回答收到底部
   - 第 3 步：在底部选择一个分析角度，让其他 AI 来审视它

2. **路由器区默认折叠**：在用户还没有引用任何内容之前，路由器区显示为折叠状态，标题旁提示"需要先引用回答才能使用"
3. **评审区默认折叠**：在还没有候选答案（没有点击过"纳入本轮"）之前，评审区折叠，标题旁提示"需要先纳入候选答案"

4. **模式/标签改为单选按钮组**（而不是下拉框），每个选项旁边直接写解释：
   - `○ 评分评审` —— 让评委 AI 给每条答案打分，按综合得分排名
   - `○ 讨论评审` —— 让评委 AI 围绕每条答案展开讨论，输出文字评价
   - `○ 匿名（A/B/C/D）` —— 评委只看编号，不知道是哪家 AI 的答案，避免偏见
   - `○ 真实模型名` —— 评委能看到每条答案来自哪个模型

**优点**：
- 用户按自然流程探索，不会被一开始的复杂界面吓到
- 选项旁边的解释消除了"不知道选哪个"的焦虑
- 折叠减少了初始界面的信息密度

**缺点**：
- 改动量中等，需要改 HTML 结构 + 少量 JS 逻辑（折叠/展开状态管理）
- 需要新增首次使用标记的 storage 逻辑
- 老用户可能会觉得折叠麻烦（可以加一个"展开全部"的开关解决）

---

### 方案 C：向导式新手教程（较大改动）

**核心思路**：做一个 step-by-step 的交互式引导（类似产品 tour），用高亮框 + 箭头指向每个 UI 区域，逐步讲解。

**具体改动**：

1. 用半透明遮罩 + 高亮框实现引导步骤（约 5-6 步）
2. 每一步解释一个区域的功能，用户可以点"下一步"或"跳过"
3. 引导完成后不再出现（除非手动重新触发）
4. 引导文案示例：

> **Step 1：全局提问区** —— 在这里输入你的问题，勾选想咨询的 AI 模型，点击"同时提问"，问题会同时发给所有勾选的 AI。
>
> **Step 2：模型卡片区** —— 每个 AI 的回答会实时显示在这里。状态灯表示 AI 是否还在生成中。
>
> **Step 3：轮次信息** —— 每次群发提问会自动创建一个"轮次"，用来记录这一轮的讨论。点击"纳入本轮"可以把某个回答标记为本轮候选。
>
> **Step 4：观点互评区** —— 这是插件的核心功能。"引用"某个 AI 的回答后，选择一个分析角度（如"找漏洞"），让其他 AI 来审视这份回答。
>
> **Step 5：评审区** —— 当你收集了足够的候选答案后，在这里启动多模型评审，让评委 AI 打分或讨论，最终得到排名和综合评价。

**优点**：
- 最完整的新手引导体验
- 用户不会遗漏任何功能
- 可以做成很精美的交互

**缺点**：
- 开发成本最高，需要新增完整的 tour 组件（遮罩、高亮定位、步骤状态机）
- 可能与现有布局产生定位冲突
- 对于只想快速使用的用户反而是干扰
- 维护成本高：UI 改动后 tour 的高亮位置可能需要同步调整

---

### 方案 D：示例视频 / 动图 + 简短文案

**核心思路**：在插件面板顶部（或折叠区内）放一个简洁的使用示例卡片，用文字描述 1-2 个典型使用场景。

**具体改动**：

1. 面板顶部新增一个可关闭的"新手提示"卡片：

```
💡 快速上手：

1. 输入问题，勾选 AI 模型，点"同时提问"
2. 看到回答后，点"引用"把想审查的回答收到底部
3. 在底部选一个分析角度（如"找漏洞"），让其他 AI 来审阅它
4. 积累多个回答后，可启动"评审"让 AI 相互打分排名

[不再显示]
```

2. 在路由器和评审区各加一行灰色小字说明该区的用途

**优点**：
- 实现成本极低（纯 HTML）
- 可以快速关闭，不干扰熟练用户
- 用真实场景解释比抽象术语更容易理解

**缺点**：
- 仅仅靠文字可能仍有用户不会读
- 没有解决"路由器"这个名称的问题（不过如果改了名字会好很多）

## 关于"路由器"命名的替代方案

| 候选名称 | 理由 | 投票 |
|----------|------|------|
| 观点互评 | 直白说出功能：把观点发给别人互相评 | ⭐⭐⭐ |
| 交叉辩论 | 来自 PRD 原始概念 "Cross-Fire"，有画面感 | ⭐⭐ |
| 深度追问 | 强调"追问"的使用场景 | ⭐⭐ |
| 分析工作台 | 更中性、专业 | ⭐ |
| 引用转发 | 最直白的功能描述 | ⭐ |

推荐 **"观点互评"**——既保留了"多个 AI 互相评鉴"的核心概念，又比"路由器"或"辩论"更温和、更容易被普通用户接受。
